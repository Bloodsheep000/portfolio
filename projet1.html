Projet 1

Déploiement Automatisé GLPI sur AWS

Infrastructure as Code (IaC) avec Terraform

1 semaine

Contexte

Migration vers le Cloud (AWS)

Rôle

Ingénieur Cloud Junior

🎯 Contexte et Problématique

Situation initiale

L'entreprise souhaite moderniser son infrastructure de gestion de parc (GLPI). L'infrastructure actuelle sur site manque de flexibilité et l'installation manuelle des serveurs prend trop de temps et est sujette aux erreurs humaines.

Problématique identifiée

Comment déployer une infrastructure GLPI fiable, reproductible et sécurisée sur le Cloud public (AWS) en moins de 10 minutes, sans intervention manuelle ?

🎯 Objectifs du Projet

1

Automatisation (IaC)

Utiliser Terraform pour provisionner les ressources AWS (Instances EC2, Sécurité, Réseau) par le code.

2

Installation logicielle automatique

Utiliser des scripts `User Data` (Bash) pour installer la pile LAMP et GLPI dès le premier démarrage du serveur.

3

Sécurité Cloud

Configurer les Security Groups et les rôles IAM pour respecter le principe de moindre privilège.

🔍 Analyse et Conception

Solutions envisagées

Solution 1 : Déploiement Manuel (Console AWS)

✅ Avantages :
  • Interface graphique intuitive pour débuter.
  • Pas de code à apprendre.
❌ Inconvénients :
  • Non reproductible (clics non enregistrés).
  • Risque d'oubli d'une étape de sécurité.
  • Lent à répliquer pour plusieurs environnements (Test/Prod).
✓ Retenue

Solution 2 : Infrastructure as Code (Terraform)

✅ Avantages :
  • Déploiement ultra-rapide et identique à chaque lancement.
  • Versionning du code (Git).
  • Documentation technique auto-générée par le code.
❌ Inconvénients :
  • Courbe d'apprentissage plus élevée (HCL).

Justification du choix : L'approche IaC correspond aux standards actuels du DevOps et des services informatiques en général. Elle garantit que le serveur GLPI sera toujours installé avec la même configuration validée.

Architecture technique cible

L'architecture repose sur une instance AWS EC2 (T3.micro) sous Ubuntu 22.04 LTS. Elle est protégée par un Security Group strict (Ports 80 et 22) et s'autoconfigure via cloud-init.

⚙️ Réalisation

Méthodologie DevOps

Le projet a été découpé en écriture de code infrastructure (HCL), écriture du script de provisionning (Bash) et déploiement.

Phase 1 Infrastructure

Codage Terraform

  • Définition du Provider AWS (Region us-east-1).
  • Recherche dynamique de l'AMI Ubuntu 22.04 LTS.
  • Configuration du Security Group (Firewall Cloud).
  • Définition de l'instance EC2 T3.micro.
Phase 2 Scripting

Automatisation Bash

  • Rédaction du script `install_glpi.sh`.
  • Automatisation de l'installation LAMP (Linux Apache MariaDB PHP).
  • Création silencieuse de la base de données SQL.
  • Téléchargement et configuration des droits GLPI.
Phase 3 Déploiement

Terraform Apply

  • Initialisation (`terraform init`).
  • Planification (`terraform plan`).
  • Application et récupération de l'IP (`terraform apply`).

Mise en œuvre technique (Extraits de code)

1. Définition de l'instance (main.tf)

Utilisation d'une instance T3.micro (éligible Free Tier) et injection du script via `user_data`. 

resource "aws_instance" "srv_glpi" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t3.micro" # Optimisé pour le Cloud AWS actuel
  key_name      = "glpikey" 

  vpc_security_group_ids = [aws_security_group.glpi_sg.id]

  # Injection du script d'installation automatique
  user_data = file("install_glpi.sh")

  tags = {
    Name = "GLPI-Server-BTS"
    Project = "BTS-SIO-BLOC-6"
  }
}

2. Script d'installation automatique (install_glpi.sh)

Ce script est exécuté par AWS au premier démarrage. Il installe la stack LAMP et configure la base de données sans intervention humaine. Il créé le fichier de configuration Apache pour GLPI et redémarre le service. 

#!/bin/bash

# 1. Mise à jour du système
apt update
apt upgrade -y

# 2. Installation Apache, MariaDB et PHP (avec les extensions requises par GLPI)
apt-get install -y apache2 mariadb-server php libapache2-mod-php php-mysql php-xml php-mbstring php-curl php-gd php-intl php-ldap php-apcu php-zip php-bz2

# 3. Configuration de la base de données (Création auto user/base)
# On lance le service
systemctl start mariadb
systemctl enable mariadb

# On créé la BDD et l'utilisateur (Mot de passe: BTSsio2024!)
mysql -e "CREATE DATABASE glpidb;"
mysql -e "CREATE USER 'glpi_user'@'localhost' IDENTIFIED BY 'admin123';"
mysql -e "GRANT ALL PRIVILEGES ON glpidb.* TO 'glpi_user'@'localhost';"
mysql -e "FLUSH PRIVILEGES;"

echo "Configuration de PHP..."
sed -i 's/memory_limit = .*/memory_limit = 256M/' /etc/php/8.1/apache2/php.ini
sed -i 's/upload_max_filesize = .*/upload_max_filesize = 20M/' /etc/php/8.1/apache2/php.ini
sed -i 's/max_execution_time = .*/max_execution_time = 300/' /etc/php/8.1/apache2/php.ini
sed -i 's/session.cookie_httponly =.*/session.cookie_httponly = on/' /etc/php/8.1/apache2/php.ini

# 4. Téléchargement et installation de GLPI (Version 10.0.10 stable)
cd /tmp
wget https://github.com/glpi-project/glpi/releases/download/10.0.15/glpi-10.0.15.tgz
tar -xzvf glpi-10.0.15.tgz -C /var/www/

# 5. Permissions (Donner les droits à Apache)
chown -R www-data:www-data /var/www/glpi
chmod -R 755 /var/www/glpi

# 6. Configuration Apache (VirtualHost basique)
cat > /etc/apache2/sites-available/glpi.conf <<'EOF'


    DocumentRoot /var/www/glpi/public

    # Si vous voulez placer GLPI dans un sous-dossier, utilisez un Alias
    # DocumentRoot /var/www
    # Alias "/glpi" "/var/www/glpi/public"

    
        Require all granted

        RewriteEngine On

        # S'assurer que les en-têtes d'autorisation sont passés à PHP
        # Nécessaire pour l'API, CalDAV, etc.
        RewriteCond %{HTTP:Authorization} ^(.+)$
        RewriteRule .* - [E=HTTP_AUTHORIZATION:%{HTTP:Authorization}]

        # Rediriger toutes les requêtes vers le routeur GLPI, sauf si le fichier existe
        RewriteCond %{REQUEST_FILENAME} !-f
        RewriteRule ^(.*)$ index.php [QSA,L]
    

    # Logs
    ErrorLog ${APACHE_LOG_DIR}/glpi_error.log
    CustomLog ${APACHE_LOG_DIR}/glpi_access.log combined

EOF

# 7. Activation du site et redémarrage
a2ensite glpi.conf
a2dissite 000-default.conf
a2enmod rewrite
systemctl restart apache2
systemctl reload apache2

3. Sécurité réseau (network_security.tf)

Configuration du pare-feu AWS (Security Group) pour n'autoriser que le strict nécessaire. 

resource "aws_security_group" "glpi_sg" {
  name = "glpi-security-group"
  
  ingress { # HTTP pour l'accès web
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress { # SSH pour l'administration
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

Difficultés rencontrées et solutions

Difficulté : Versions AMI Ubuntu

Problème : Le code Terraform ne trouvait pas l'image Ubuntu 24.04 à cause d'un changement de nommage sur AWS.

Solution : Passage à la version 22.04 LTS (plus stable pour GLPI) et ajustement des filtres de recherche Terraform.

Difficulté : Type d'instance AWS

Problème : L'instance `t2.micro` n'était plus disponible dans l'offre gratuite dans ma région.

Solution : Modification du code Terraform pour utiliser `t3.micro`.

🚀 Perspectives et Améliorations

Évolutions possibles

  • Séparer la base de données sur un service managé (AWS RDS) pour la redondance.
  • Stocker l'état Terraform (tfstate) dans un bucket S3 pour le travail en équipe.

Ce que je referais différemment

J'utiliserais Ansible en complément de Terraform. Terraform créerait le serveur, et Ansible se chargerait de la configuration fine de GLPI, ce qui est plus flexible qu'un script Bash unique.
